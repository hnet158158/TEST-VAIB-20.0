# ОБЯЗАТЕЛЕН к прочтению для Coder (Vaib4), Editor (Vaib6), Tester (Vaib5) и Archaeologist (Vaib99).

## 1. 🗂️ СЕМАНТИЧЕСКИЕ БЛОКИ (SEMANTIC REGIONS)
Эти теги используются для структурирования файлов на высоком уровне. Язык комментариев зависит от языка программирования (например, `#` для Python, `//` для JS/TS).

- **Заголовок модуля (Module Header):**
  Помещается в самом начале файла.
  `# START_MODULE_CONTRACT` ... `# END_MODULE_CONTRACT`
  `# START_MODULE_MAP` ... `# END_MODULE_MAP`

- **Логические блоки (Logical Blocks):**
  Оборачивают классы или группы связанных функций.
  `# START_BLOCK_<NAME>` ... `# END_BLOCK_<NAME>`

## 2. ⚓ ФУНКЦИОНАЛЬНЫЕ КОНТРАКТЫ (GRACE ANCHORS)
Контракты объясняют агентам смысл (Intent) кода. Чтобы не сбивать паттерны предобучения LLM, мы используем нативные форматы документации, но со строгими правилами.

**Правила написания (Анти-дублирование):**
- **Input / Output:** ЗАПРЕЩЕНО указывать типы данных (LLM видит их в сигнатуре). Пишите ТОЛЬКО бизнес-ограничения (например, "id должен быть активным", "гарантированно не бросает throw").
- **Intent:** Описание бизнес-цели на РУССКОМ языке (Что и зачем делает).

**Вариант А: Python (Контракт ВНУТРИ)**
В Python контракты живут строго внутри функции в виде нативных Docstring:
```python
def process_payment(amount: float, user_id: int) -> bool:
    """
    [START_CONTRACT_PROCESS_PAYMENT]
    Input: amount - строго больше нуля; user_id - валидный ID пользователя.
    Intent: Провести транзакцию и обновить баланс.
    Output: Успешность операции, гарантированно без броска исключений.
    [END_CONTRACT_PROCESS_PAYMENT]
    """
    ...
```

**Вариант Б: TS / JS / Java (Разделение контракта и якоря)**
В C-подобных языках контракт пишется снаружи (как JSDoc), а внутрь тела ставится неудаляемый маркер для ИИ-редактора:
```typescript
/**
 * [START_CONTRACT_PROCESS_PAYMENT]
 * Input: amount - строго больше нуля; userId - валидный ID пользователя.
 * Intent: Провести транзакцию и обновить баланс.
 * Output: Успешность операции, гарантированно без throw.
 * [END_CONTRACT_PROCESS_PAYMENT]
 */
function processPayment(amount: number, userId: number): boolean {
    // AI_ANCHOR_PROCESS_PAYMENT
    ...
}
```

## 3. ⚡ COMPACT MODE (ДЛЯ ТРИВИАЛЬНЫХ ФУНКЦИЙ)
Для геттеров, сеттеров и простых утилит (до 3 строк кода) полноформатные контракты `START_CONTRACT` ЗАПРЕЩЕНЫ, чтобы не засорять контекст. Используйте однострочный формат:
`# CONTRACT: Intent: Конвертация даты в ISO. Output: Строка формата YYYY-MM-DD.`

## 4. 📝 СИСТЕМА ЛОГИРОВАНИЯ (BELIEF SYSTEM)
Мы используем концепцию [Belief] логов для автоматизированной верификации и трассировки (работает как Chain-of-Thought для ИИ). Агент-тестировщик читает эти логи, чтобы подтвердить, что реальность совпадает с контрактом.

Обязательный шаблон (вставляется внутри функции сразу после контракта или якоря):
`logger.debug("[Module][Function] Belief: <Intent> | Input: <Args> | Expected: <Postcondition>")`

## 5. 🛠️ ПРАВИЛА РЕДАКТИРОВАНИЯ И ИНЖЕКТА (ДЛЯ EDITOR И ARCHAEOLOGIST)
1. **Синхронизация контрактов (Contract Sync):** Если логика функции изменяется (рефакторинг/багфикс), вы ОБЯЗАНЫ обновить текст `Intent` и `Output` внутри тегов `START_CONTRACT`. Контракт всегда должен соответствовать реальности.
2. **Целостность тегов (Integrity):** ЗАПРЕЩЕНО удалять или выносить теги `START_CONTRACT` или `AI_ANCHOR` за пределы их предписанных мест. Якорь `AI_ANCHOR` внутри TS/JS функций критически важен для парсера.
3. **Легаси-код (Legacy Integrity):** При разметке старого кода (Archaeologist) разрешается ТОЛЬКО добавлять новые GRACE-анкоры. Категорически ЗАПРЕЩЕНО переводить существующие английские комментарии разработчиков.
